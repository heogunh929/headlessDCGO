## 문제 (AS-IS / TO-BE 이전)

- 원본은 패배 신호를 발생 지점에서 `Player.IsLose`로 적립하고(`AttackProcess`의 `SecurityCards.Count == 0`, `TurnStateMachine`의 `LibraryCards.Count == 0`), 공통 처리(`AutoProcessing`)가 `player.IsLose`를 읽어 게임 종료를 확정한다.
- TO-BE의 `PlayerRuleAdapter`는 terminal 조건(`EvaluateDeckLossOnDraw`/`EvaluateSecurityAttack`/`EvaluateLoseFlag`)을 계산하지만 `DcgoMatch`/`InMemoryRuleQueryService.SetTerminal()`과 연결되지 않았다.
- `GameFlowProcessor.EndTurnCheck`는 placeholder(`return false`)였고, `ITerminalStateController.SetTerminal(bool)`은 승자/사유를 담지 못해 `SetTerminal(true)`만으로는 `MatchResult.WinnerId`가 채워지지 않았다(deck-out 경로가 이를 증명).

## 구현 요약

- **신호 적립(이벤트 지점) / 판정(공통 루프) 분리** — AS-IS 모델 충실 재현:
  - `AttackPipeline.AdvanceCombatAsync`: 직접 공격(`IsDirectAttack && TargetId==null && !IsBlocked`)에서 `SecurityResolver`가 빈 시큐리티(`DefenderHasNoSecurity`)로 실패하면 → `PlayerStatusController.MarkLose(defender)`. 시큐리티가 존재해 흡수된 경우(트래시 1장)는 lose가 적립되지 않으므로 "1장 흡수 후 0장" 오탐이 발생하지 않음.
  - `GameFlowProcessor.EndTurnCheck`: 이미 terminal이면 no-op. 아니면 `TerminalEvaluator.Evaluate`로 판정 → terminal 시 `ITerminalOutcomeSink.SetTerminalOutcome(winner, isDraw:false, message)`.
- `TerminalEvaluator`: `TurnController.Current.PlayerOrder`로 `MatchState`(플레이어 + lose 플래그만, 카드 zone 불필요)를 구성하고 `MemoryController.Current`로 `PlayerRuleAdapter`를 만들어 `EvaluateLoseFlag`를 플레이어별로 평가, 첫 terminal 판정을 반환(승자=상대).
- `InMemoryRuleQueryService`: `SetTerminalOutcome`이 `_isTerminal=true` + outcome 저장. `SetTerminal(false)`/`ResetMatchState`는 outcome도 정리.
- `DcgoMatch.StepAsync`: terminal이면서 action metadata로 `MatchResult`가 안 채워진 경우 룰 쿼리 서비스의 `TryGetTerminalOutcome`을 읽어 `MatchResult(winner, isDraw, reason)`로 승격 → 기존 action-metadata 경로(`ApplyTerminalResultMetadata`)와 공존.
- `EngineContext`: `PlayerStatusController`를 마지막 옵션 생성자 파라미터로 추가(기존 호출부 무영향), 서비스 등록 + `ResetMatchState` 연결, `CreateDefault` 기본 인스턴스 주입.

## 설계 메모 (설계안 대비)

- 설계안 2.5는 `EndTurnCheck = X-02`로 매핑한다. 직접 공격의 패배 신호는 타이밍 민감(흡수 vs 직격)하므로, 발생 지점(`AttackPipeline`)에서 lose 플래그로 적립하고 `EndTurnCheck`가 `PlayerRuleAdapter`로 판정/결과배선하는 **AS-IS `AutoProcessing` 패턴**을 따랐다. lose 플래그가 내구 신호이므로 공격 정리(Resolved→Completed→None)와 무관하게 같은 루프 반복에서 정확히 판정된다.

## 테스트하지 못한 항목과 이유

- Deck-out terminal은 기존 `HeadlessEarlyPhaseFlow`가 draw 시점에서 직접 `SetTerminal(true)`로 처리하던 경로를 **그대로 유지**(G2A-003 회귀 보호). 본 Goal은 이를 lose 플래그로 일원화하지 않고 공존시킴 — 통합 일원화는 G3.5-009/Phase 4에서 대표 카드와 함께 검토.
- surrender/draw(무승부) outcome은 sink 인터페이스/`MatchResult`가 지원하나 자동 판정 경로는 본 Goal에서 다루지 않음(명시 action-metadata 경로가 이미 커버).
- 실제 카드 효과가 lose 플래그를 부여하는 end-to-end 경로는 카드 풀이 있는 G3.5-009 E2E/Phase 4에서 대표 카드로 검증 예정.

## 미해결 리스크

- 현재 자동 패배 판정은 직접 공격(시큐리티 0)과 외부 lose 플래그(향후 카드 효과/항복)만 커버. memory-out 등 추가 종료 규칙은 별도 Goal에서 신호 적립부를 추가하면 동일 `EndTurnCheck` 경로로 수렴한다.
- `TerminalEvaluator`는 2인전 기준으로 승자=상대를 산출(`PlayerRuleAdapter.OpponentOf`). 다인전 확장은 Phase 4에서 별도 검토.
