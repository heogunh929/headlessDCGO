## Goal 범위

## 문제 (AS-IS / TO-BE 이전)

- 원본 `ContinuousController`는 modifier/restriction/replacement를 상시 재평가하여 공격/블록 가능 여부에 반영한다.
- TO-BE의 `ContinuousEffectEvaluator` + `RestrictionHelpers`는 contract + test(G3I-002/G3H-002)로 완비되어 있으나, `HeadlessLegalActionDispatcher` / `AttackPermanentAction` / `BlockTiming`이 이 출력을 전혀 사용하지 않았다.
- 공격/블록 적법성은 `CardInstanceRecord`/`CardRecord`의 **정적 metadata 불리언**(`cannotAttack`/`cannotBlock` 등)만으로 판정 → 다른 카드의 지속효과로 부여되는 "공격 불가/블록 불가"가 legal action에 반영되지 않음.

## 구현 요약

- `ContinuousRestrictionGate` (X-04 브리지):
  - `Evaluate(context, entityId)`: `ContinuousEffectEvaluator.Evaluate(context.EffectRegistry, EffectQueryContext("ContinuousRecalculation", targetEntityId: entityId))`로 **레지스트리의 continuous-role 효과만** 조회 → `result.Restrictions` 반환. (카드/인스턴스 metadata 제약은 기존 정적 검사가 이미 담당하므로 중복 조회하지 않음)
  - `EvaluateAttack`/`EvaluateBlock`: `RestrictionHelpers.CannotAttack`/`CannotBlock`로 제약 평가.
  - 기본 `EffectRegistry`가 비어 있으므로 지속효과 등록 전에는 완전한 no-op → Phase 4 카드 풀 등록 시 그대로 동작.
- `AttackPermanentAction.Validate`: 기존 정적 `cannotAttack` 검사 직후 `ContinuousRestrictionGate.EvaluateAttack`를 호출, 제약 시 `Illegal(reason)`. `GetLegalActions`/`Process`가 동일 `Validate`를 쓰므로 목록 생성과 실행이 일관되게 제외/거부된다.
- `BlockTiming.TryCreateCandidate`: 기존 정적 `cannotBlock` 검사 직후 `ContinuousRestrictionGate.EvaluateBlock(context, blockerId, attack.AttackerId)` 제약 시 후보에서 제외 → block choice 후보/선택 검증에 동시 반영.

## 설계 메모 (설계안 대비)

- 설계안 2.5는 continuous 반영을 `RuleProcessAsync`에 배치했으나, X-04의 실제 갭("지속효과가 **legal action**에 반영 안 됨")은 legal action이 매 요청마다 즉석 생성되는 구조상 **후보 생성기(`AttackPermanentAction`/`BlockTiming`)** 가 정확한 통합 지점이다(감사 보고서 4.5/결론과 일치). `RuleProcessAsync`(DP 음수 삭제 등 상태 정리)는 별개 관심사로 placeholder 유지.

## 테스트하지 못한 항목과 이유

- Restriction-role 바인딩(`GetRestrictionEffects`) 경로는 본 게이트가 continuous-role(`GetContinuousEffects`)만 조회하므로 직접 반영하지 않는다. 해당 경로는 helper 단위테스트(G3H-002)가 커버하며, 카드 포팅 시 continuous-role로 등록하는 규약으로 수렴한다.
- modifier(DP 증감) → battle 결과 반영은 X-03 범위로 본 Goal에서 다루지 않음(현재 `BattleResolver`는 metadata DP 기반).
- 실제 카드의 지속효과가 등록되어 end-to-end로 동작하는 경로는 카드 풀이 있는 Phase 4 및 G3.5-009 E2E에서 대표 카드로 검증 예정.

## 미해결 리스크

- 게이트 scope는 evaluator 단위테스트와 동일한 `"ContinuousRecalculation"` 문자열을 사용한다. 카드 포팅 시 지속효과 binding이 이 scope/role로 등록되어야 반영된다(명명 규약은 Phase 4에서 binding 등록 규칙과 함께 확정).
- 현재 공격 제약은 attacker 단위(general)로 평가한다. target/source 한정 제약(예: "특정 색만 공격 불가")은 `CannotRestriction.SourceEntityId`/availability로 표현 가능하나 광범위 케이스 E2E는 Phase 4/G3.5-009로 확장.
