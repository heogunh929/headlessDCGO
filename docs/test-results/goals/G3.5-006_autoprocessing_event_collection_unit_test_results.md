## 문제 (AS-IS / TO-BE 이전)

- 원본 `AutoProcessing.AutoProcessCheck()`는 `StackSkillInfos(null, RulesTiming)`로 상태 변화에서 타이밍 스킬을 적재하고 `TriggeredSkillProcess`로 해결한다(이벤트 구동 트리거 수집).
- TO-BE의 `AutoProcessingTriggerCollector`/`EndAttackTriggerHook`/`SecurityDelayedTriggerHook`은 contract + 테스트로만 존재했고, `GameFlowProcessor.AutoProcessAsync`는 `EffectScheduler.ResolveAllAsync`만 호출(이미 적재된 효과만 배수)했다.
- 게임 이벤트는 `DcgoMatch._pendingEvents`, `InMemoryZoneMover._events`, `MatchState.Events`에 **분산 저장**되어 있을 뿐, 트리거 수집을 구동하는 단일 소스/버스가 없었다. (`AttackPipeline`의 end-attack 경로만 부분적으로 collector를 호출)

## 구현 요약

- `GameEventQueue`: 트리거 수집용 **단일 drainable 이벤트 소스**.
  - `Publish(GameEvent)`: 컴포넌트/테스트가 합성 트리거 이벤트를 직접 적재.
  - `SyncFrom(IReadOnlyList<GameEvent>)`: append-only 이벤트 로그(`IZoneMover.Events`)에서 **미소비분만** 적재. 매치 내 source는 append-only이므로 count 기반 커서로 충분하며, 큐 리셋과 함께 커서도 0으로 초기화 → 동일 이동의 중복 수집 방지.
  - `DrainPending()`: publish 순서대로 전량 반환·비움.
- `GameFlowProcessor.AutoProcessAsync` (X-05):
  1. `GameEventQueue.SyncFrom(context.ZoneMover.Events)` — 카드 이동을 자동으로 트리거 윈도우로 전환.
  2. 큐 drain → 각 이벤트를 `AutoProcessingTriggerCollector.CollectAndEnqueue(event, scheduler)`로 수집·적재(`Unknown` 이벤트는 스킵).
  3. `EffectScheduler.ResolveAllAsync` — 수집된 트리거 + 기타 적재 효과 해결.
  - 반환을 `(Resolved, Collected)`로 분리해, `ResolvedEffectCount`(관측/트레이스용)는 실제 해결 수만 집계하고 진행(progress) 판정은 `resolved>0 || collected>0`로 한다. (수집·해결이 다른 pass에 걸쳐도 루프가 진행을 감지)
- 빈 `EffectRegistry`(기본값)에서는 수집 0건 → 진행 0 → 루프가 즉시 안정 상태로 종료(무한 루프 없음). 카드풀이 들어오는 Phase 4에서 timing별 binding이 채워지면 그대로 동작한다.

## 미해결 리스크

- ZoneMover 브리지는 매 step `ZoneMover.Events` 전량을 커서 이후만 동기화한다. 효과 해결이 자기 timing과 일치하는 카드 이동을 다시 유발하면 재트리거가 가능하나, `GameFlowProcessor.MaxIterations`(256) 가드로 무한 루프를 방지한다. 카드 설계 차원의 정합성은 Phase 4에서 엄격히 검증한다.
- 현재는 `CardMoved`/`StateChanged` 등 raw 이벤트 타입을 timing 문자열로 사용(메타데이터 `triggerTiming` 우선). 타이밍 명명 규약 정합(예: `OnPlay` 매핑)은 Phase 4 카드 포팅 시 binding 등록 규칙과 함께 확정한다.
