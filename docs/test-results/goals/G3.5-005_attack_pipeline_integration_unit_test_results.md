## 구현 요약

- `AttackPhase` 도입: AS-IS `AttackProcess.ProcessNextState()`의 Counter/Block/Battle/Security/End/CleanUp를 headless phase로 매핑.
  - `None → Declared → Blocking → Combat → Resolved → Completed → (clear) None`
- `AttackPipeline.AdvanceAsync(context, ct)`는 현재 `Phase`에 따라 **정확히 1단계** 진행:
  - **Declared**: `BlockTiming.RequestBlockChoice`. 후보가 있으면 choice를 열고 `Blocking`으로 전이(루프는 다음 반복 진입 시 pending choice로 중단). 후보가 없으면(auto-skip) 곧장 `Combat`.
  - **Blocking**: block choice가 해결된 후 재개 지점. `Combat`으로 전이(선택 시 `SelectBlocker`가 이미 적용, skip 시 원래 타깃 유지).
  - **Combat**: direct(타깃 없음·미블록)면 `SecurityResolver`, 그 외는 `BattleResolver`. resolver가 `ResolveAttack` 호출 → `Resolved`. resolver 실패 시 무한 루프 방지를 위해 강제 `ResolveAttack` + `LogSink.Warn`.
  - **Resolved**: `EndAttackTriggerHook.Process`로 OnEndAttack 트리거 수집/적재 → `Completed`.
  - **Completed**: `ClearAttack` → `None`.
- `GameFlowProcessor`는 `Phase != None`인 동안 매 반복 attack을 1단계 진행하므로, AutoProcess(트리거 효과 배수)와 교차하며 안정 상태까지 반복한다.
- block choice 라우팅: 게임 루프가 pending choice로 중단된 뒤 `ResolveChoice` 액션이 들어오면, `MetadataActionProcessor`가 `ChoiceType.Blocker`를 감지해 `BlockTiming.ResolveBlockChoice`로 보낸다. 이로써 일반 `ResolveChoice`가 attack 상태를 갱신하지 못하던 누락(선택 blocker가 `SelectBlocker`로 반영되지 않던 문제)을 해소한다.

## 기존 테스트 적응 사유

- 게임 루프가 이제 공격을 자동 진행하므로, "루프로 공격을 선언한 뒤 sub-resolver를 직접 호출해 중간 상태를 검사"하던 Phase 2 단위테스트들은 선언 직후 상태를 관찰할 수 없게 되었다.
- 이들 테스트의 본래 의도(개별 resolver/declare/end-turn cleanup 격리 검증)를 유지하기 위해, 공격을 게임 루프 대신 **`AttackController.DeclareAttack` 또는 `AttackPermanentAction.Process`로 직접 셋업**하도록 조정했다. 단언 항목은 동일하게 유지했다.
- 통합 동작(자동 진행/중단·재개) 자체는 신규 `G3.5-005` E2E 테스트가 검증한다.

## 미해결 리스크

- direct/target 판정은 `IsDirectAttack`/`TargetId`/`IsBlocked` 조합으로 처리한다. AS-IS의 "direct 공격도 block 가능(blocker로 redirect)" 규칙은 BlockTiming이 후보를 제공하므로 동작하나, redirect 후 security가 아닌 battle로 전환되는 세부 케이스의 광범위 E2E는 G3.5-009에서 확장한다.
- resolver 검증 실패 시 강제 `ResolveAttack`로 루프 안정성을 보장하지만, 이는 방어적 처리이며 실제 비정상 입력은 Phase 4 카드 포팅 단계에서 더 엄격히 검증한다.
