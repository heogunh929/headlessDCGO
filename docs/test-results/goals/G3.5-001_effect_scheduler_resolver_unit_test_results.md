## Goal 범위

- 목표: EffectScheduler에 카드효과 resolver 연결 (B-03 수정)
- 작업 범위: 큐 전용 `EffectScheduler`가 no-op 대신 실제 `IHeadlessCardEffect` 본문을 실행하도록 배선

## 문제 (AS-IS)

- `EffectScheduler.DefaultResolverAsync`가 입력과 무관하게 항상 `EffectResult.Success()`를 반환해 카드효과 본문이 한 번도 실행되지 않았다 (B-03).
- `EffectBinding`은 `EffectRequest`만 보존하고 `ToBinding` 시점에 `IHeadlessCardEffect` 본문을 폐기해, resolver가 본문을 찾을 경로 자체가 없었다.
- `EngineContext`는 `EffectRegistry`를 보유하지 않고 기본 `new EffectScheduler()`(no-op resolver)만 구성했다.

## 구현 요약

- `EffectBinding`에 선택적 `IHeadlessCardEffect? Effect`를 추가했다. 본문이 첨부되면 `effect.Definition.EffectId == request.EffectId`를 검증한다.
- `KeywordBaseBatch1/2.ToBinding`이 생성한 effect 본문을 binding에 첨부하도록 했다 (대표 경로).
- `CardEffectSchedulerResolver.Create(registry, resolver?, sinkFactory?)`가 `Func<EffectRequest, CancellationToken, Task<EffectResult>>`를 만들어 `EffectScheduler`에 주입한다.
  - `registry.Find(request.EffectId)?.Effect`가 있으면 `HeadlessCardEffectResolver`로 실행하고 mutation sink를 주입한다.
  - 기본 sink는 `RecordingEffectMutationSink`이며 결과 values에 `mutationCount`를 기록한다.
  - 본문이 미연결이면 점진적 통합을 위해 `unresolved=true` 메타데이터와 함께 success no-op을 반환해 큐 배수를 막지 않는다.
- `EngineContext`에 `EffectRegistry` 프로퍼티/서비스 등록을 추가하고, `CreateDefault`가 `InMemoryEffectRegistry` + `CardEffectSchedulerResolver` 기반 `EffectScheduler`를 구성하도록 했다.
- 기존 호출부 호환을 위해 `EffectBinding` 생성자와 `EngineContext` 생성자에 추가한 파라미터는 모두 선택적(기본값 null)으로 두었다.

## 미해결 리스크

- 현재 mutation은 수집(record)만 되고 게임 상태에 적용되지 않는다 → G3.5-002에서 production sink로 교체 필요.
- 미연결 effect를 success no-op으로 흘리는 정책은 점진적 통합을 위한 임시 동작이며, 전체 effect 연결 완료 후(E2E, G3.5-009) 미연결을 오류로 강화할지 재검토한다.
